# CLAUDE.md — lwc-tailwind

## What this project is

A Salesforce CLI plugin (`sf` CLI) that enables Tailwind CSS in Lightning Web Components (LWC). LWC uses Shadow DOM which blocks regular CSS — this plugin works around it by splitting Tailwind output into per-component CSS files and a shared static resource for base variables.

## Architecture

```
src/tailwind.css                          ← Tailwind source (user's project)
    │  (PostCSS)
    ▼
src/.tailwind-compiled.css                ← Full compiled output (intermediate, not deployed)
    │  (parseTailwindCss)
    ├─→ staticresources/tailwind.css      ← Base variables only (--tw-* custom properties)
    └─→ lwc/<component>/<component>.css   ← Per-component utility rules (matched from .html + .js)
```

- **Base variables** (`--tw-*`): Stored in a single static resource, injected into each shadow root via `loadStyle` by the `TailwindElement` base class
- **Utility rules**: Split per-component — each LWC gets only the Tailwind classes it uses, appended below a `/* === GENERATED BY TAILWIND SPLITTER === */` marker in its `.css` file
- Hand-written CSS above the marker is preserved

```

```

## Key concepts

- **Shadow DOM isolation**: Regular CSS can't cross shadow boundaries. CSS custom properties (variables) can. That's why `--tw-*` vars go in a shared static resource and utility classes go per-component.
- **SLDS design tokens**: The tailwind.config.js maps Salesforce design tokens (`--lwc-colorBrand`, etc.) to Tailwind colors. These are set by the Salesforce platform and penetrate shadow DOM.
- **The marker pattern**: Component CSS files have hand-written CSS above `/* === GENERATED BY TAILWIND SPLITTER === */` and generated rules below. The splitter replaces everything below the marker on each build.
- **`TailwindElement`**: Base class components extend instead of `LightningElement`. It handles loading the static resource CSS into each shadow root.

## Commands

```bash
npm run build          # Compile TypeScript → lib/
npm run clean          # Remove lib/
sf plugins link .      # Link plugin locally for testing
```

After making TypeScript changes, you MUST run `npm run build` before testing with `sf` commands. The `sf` CLI uses the compiled `lib/` directory and shows a warning about "existing compiled source" — this means stale JS will be used if you don't rebuild.

## Testing changes

```bash
npm run build && sf plugins link .
cd example
sf tailwind build                  # Should show base vars + per-component split
sf tailwind build --production     # Minified output
sf tailwind component testWidget   # Creates a new component
sf tailwind watch                  # Starts watcher (Ctrl+C to stop)
```

Verify after build:
- `example/force-app/main/default/staticresources/tailwind.css` should contain ONLY `--tw-*` variable definitions (~105 lines)
- Component `.css` files should contain only utility class rules (no base variable block)

## Important gotchas

- **Stale compiled JS**: The biggest source of bugs during development. Always `rm -rf lib && npm run build` after TypeScript changes before testing with `sf` commands.
- **Component names**: LWC requires camelCase (e.g., `myButton`, not `my-button`)
- **`postcss-cli`**: Required at runtime in user projects — it's in the `USER_DEPS` array in `init.ts`
- **The `compiledCssPath`** (`src/.tailwind-compiled.css`): Intermediate file, should never be deployed or committed. The `init` command adds it to `.forceignore` and `.gitignore`.

## CSS splitting logic (css-builder.ts)

1. `compileCss()` — Runs `npx postcss` to compile Tailwind
2. `parseTailwindCss()` — Separates base variable rules from utility class rules using `isBaseRule()` (checks: no class selectors + all decls start with `--tw-`)
3. `splitCss()` — Writes base vars to static resource, then for each LWC component:
   - Scans `.html` for `class="..."` attributes
   - Scans `.js` for string literals containing Tailwind class names
   - Matches found classes against the parsed CSS rules
   - Writes matched rules to the component's `.css` file (preserving hand-written CSS above the marker)
